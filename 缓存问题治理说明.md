# 缓存问题治理说明

## 1. 文档目的

本文档专门说明本项目中是如何处理下面三个经典缓存问题的：

1. 缓存穿透
2. 缓存击穿
3. 缓存雪崩

说明重点包括：

- 为什么会出现这些问题
- 本项目采用了什么解决方案
- 请求在代码里是按什么步骤执行的
- 涉及到的具体函数、常量、配置项分别负责什么
- 这些方案对应的知识点是什么
- 当前实现还存在哪些边界和后续可优化点

本文档对应的核心代码主要位于：

- `product_service/api/routes.py`
- `product_service/config.py`

商品详情接口是本次缓存治理的主战场：

- `GET /api/products/{product_id}`

因为它是最典型的高频读接口，也是最容易在压测中暴露缓存问题的地方。

---

## 2. 先看整体思路

本项目没有把三类问题拆成三套完全独立的逻辑，而是把它们融合在商品详情查询链路中，形成了一条完整的读路径。

整体思路如下：

1. 先查 Redis 商品详情缓存
2. 如果命中正常缓存，直接返回
3. 如果命中空值缓存，直接判定商品不存在
4. 如果缓存未命中，先查 Redis 中维护的商品 ID 索引
5. 如果索引就表明商品不存在，直接拦截，并写入空值缓存
6. 如果商品可能存在，再尝试获取 Redis 分布式锁
7. 抢到锁的请求负责查数据库并重建缓存
8. 没抢到锁的请求短暂等待缓存回填
9. 如果等待超时，再执行一次兜底查库
10. 写缓存时使用随机 TTL，避免大量 Key 同时过期

这条读链路实际上同时做了三件事：

- 用“商品 ID 索引 + 空值缓存”处理缓存穿透
- 用“分布式锁 + 等待回填”处理缓存击穿
- 用“随机 TTL”处理缓存雪崩

## 4. 关键常量设计

这些常量定义在 `product_service/api/routes.py` 中：

```python
PRODUCT_CACHE_PREFIX = "product:"
PRODUCT_LOCK_SUFFIX = ":lock"
PRODUCT_EMPTY_PLACEHOLDER = "__PRODUCT_NOT_FOUND__"
PRODUCT_ID_SET_KEY = "product:ids"
PRODUCT_ID_INDEX_READY_KEY = "product:ids:ready"
PRODUCT_ID_INDEX_LOCK_KEY = "product:ids:lock"
```

它们的含义如下：

- `PRODUCT_CACHE_PREFIX`
  - 商品详情缓存前缀
- `PRODUCT_LOCK_SUFFIX`
  - 商品详情锁后缀
- `PRODUCT_EMPTY_PLACEHOLDER`
  - 空值缓存占位符，表示“这个商品不存在”
- `PRODUCT_ID_SET_KEY`
  - Redis 中维护的商品 ID 集合
- `PRODUCT_ID_INDEX_READY_KEY`
  - 标记商品 ID 索引是否已经初始化完成
- `PRODUCT_ID_INDEX_LOCK_KEY`
  - 初始化商品 ID 索引时使用的锁

### 为什么要单独定义常量

因为缓存治理不是只有一处读写，后面很多函数都共享同一套协议。如果这些 Key 规则直接散落在各个函数内部：

- 容易拼错
- 维护困难
- 以后压测、日志排查、手动清缓存都不方便

所以这里把它们集中定义，相当于先把“缓存协议”稳定下来。

## 5. 请求主链路总览

核心入口函数：

```python
@router.get("/{product_id}", response_model=ResponseModel[schemas.ProductResponse])
def get_product(product_id: str, db: Session = Depends(database.get_db)):
```

这就是商品详情接口。后面三类缓存问题的治理，都是围绕这个函数展开的。

可以把它理解为下面这张逻辑图：

```text
请求进入
  -> 查详情缓存
      -> 命中正常缓存：直接返回
      -> 命中空值缓存：直接 404
      -> 未命中：继续
  -> 查商品 ID 索引
      -> 明确不存在：写空值缓存并 404
      -> 可能存在：继续
  -> 抢分布式锁
      -> 抢到：负责查库并重建缓存
      -> 没抢到：等待其他请求回填缓存
          -> 等到了：直接返回缓存
          -> 没等到：兜底查库
```

## 6. 缓存穿透

### 6.1 什么是缓存穿透

缓存穿透指的是：

- 请求访问的数据根本不存在
- Redis 中没有这个 Key
- 数据库里也没有这条记录
- 结果请求每次都会直接打到数据库

如果有人持续请求不存在的商品 ID，就会出现：

- Redis 一直 miss
- MySQL 被重复无意义访问
- 在高并发下数据库压力快速升高

### 6.2 本项目的处理方案

本项目对缓存穿透做了两层处理：

1. 先查 Redis 商品 ID 索引
2. 对确认不存在的商品写入空值缓存

也就是：

- 第一层尽量在“查数据库之前”把非法请求拦掉
- 第二层把“商品不存在”这个结果缓存起来，避免重复穿透

### 6.3 第一步：维护商品 ID 索引

相关函数：

- `ensure_product_id_index`
- `add_product_id_to_index`
- `remove_product_id_from_index`
- `product_may_exist`

### 6.3.1 初始化索引

函数：

```python
def ensure_product_id_index(db: Session) -> None:
```

它的逻辑是：

1. 先看 `product:ids:ready` 是否存在
2. 如果存在，说明索引已经构建过，直接返回
3. 如果不存在，就尝试获取 `product:ids:lock`
4. 抢到锁的请求负责把数据库里的所有商品 ID 全量加载到 Redis Set
5. 写入 `product:ids:ready = 1`

对应代码片段：

```python
product_ids = [row[0] for row in db.query(models.Product.id).all()]
pipeline = redis_client.pipeline()
if product_ids:
    pipeline.sadd(PRODUCT_ID_SET_KEY, *product_ids)
pipeline.set(PRODUCT_ID_INDEX_READY_KEY, "1")
pipeline.execute()
```

### 6.3.2 为什么需要这层索引

因为它能回答一个关键问题：

- “这个商品 ID 至少有没有可能存在？”

如果答案是“不可能”，那就没有必要再继续走：

- 分布式锁
- 数据库查询
- 缓存回填

也就是说，这层索引是在真正访问数据库之前，先做一次快速过滤。

### 6.3.3 为什么这里没有直接用 Bloom Filter

因为当前项目更偏课程作业和可解释性实现，Redis Set 更适合当前阶段：

- 直观
- 容易验证
- 不会引入误判
- 写法更容易讲清楚

而 Bloom Filter 虽然更省内存，但会有误判率，需要更多额外解释和实现成本。

### 6.4 第二步：判断商品是否“可能存在”

函数：

```python
def product_may_exist(product_id: str, db: Session) -> bool:
```

逻辑是：

1. 先确保索引已经准备好
2. 如果索引还没就绪，则保守放行
3. 如果索引已就绪，则执行 `sismember`

关键代码：

```python
if not redis_client.exists(PRODUCT_ID_INDEX_READY_KEY):
    return True
return bool(redis_client.sismember(PRODUCT_ID_SET_KEY, product_id))
```

### 为什么叫 `may_exist` 而不是 `exists`

因为这层索引只是工程优化手段，不是最终事实源。

一旦遇到下面这些情况：

- Redis 短暂异常
- 索引还没初始化完
- 索引数据和数据库短时不一致

系统宁可放行到数据库再查一次，也不愿把一个真实存在的商品误判成不存在。

这是一种“保守放行”的设计。

### 6.5 第三步：写空值缓存

函数：

```python
def cache_empty_product(product_id: str) -> None:
```

关键代码：

```python
redis_client.set(
    product_cache_key(product_id),
    PRODUCT_EMPTY_PLACEHOLDER,
    ex=ttl_with_jitter(
        settings.product_null_cache_ttl_seconds,
        settings.product_null_cache_ttl_jitter_seconds,
    ),
)
```

### 为什么空值缓存有效

因为它把“查不到商品”这个结果缓存了起来。

这样下一次再访问同一个不存在的商品时：

- 不需要再走数据库
- 直接在 Redis 命中空值占位符
- 立即返回 404

### 6.6 在主流程里是怎么触发的

第一种情况：命中空值缓存

```python
if cache_status == "NULL":
    raise HTTPException(status_code=404, detail="商品不存在")
```

第二种情况：索引直接判断商品不存在

```python
if not product_may_exist(product_id, db):
    cache_empty_product(product_id)
    raise HTTPException(status_code=404, detail="商品不存在")
```

第三种情况：查数据库确认商品确实不存在

```python
if not product:
    cache_empty_product(product_id)
    remove_product_id_from_index(product_id)
    raise HTTPException(status_code=404, detail="商品不存在")
```

### 6.7 对应知识点总结

- Cache Penetration
- Negative Cache
- Existence Index
- Redis Set
- 保守放行与最终一致性

### 6.8 当前边界与后续优化

- 空值缓存不是强一致方案
- 如果一个商品“先被访问不存在，后被创建”，在空值缓存过期前可能仍会返回 404
- 当前索引是 Redis Set，规模继续变大后可以考虑 Bloom Filter

## 7. 缓存击穿

### 7.1 什么是缓存击穿

缓存击穿指的是：

- 某个热点 Key 在高并发下非常常用
- 它恰好过期了
- 大量请求同时发现缓存 miss
- 然后一起冲到数据库

和穿透不同，击穿访问的是“真实存在的数据”，只是因为热点数据刚好失效，回源并发过高。

### 7.2 本项目的处理方案

本项目的击穿治理思路是：

1. 缓存失效后，不允许所有请求一起回源
2. 使用 Redis 分布式锁，保证只有一个请求负责重建缓存
3. 其他请求短暂等待缓存回填
4. 如果等待超时，再走兜底查库

这是一种比较经典的互斥重建方案。

### 7.3 第一步：尝试获取锁

函数：

```python
def acquire_lock(lock_key: str, ttl_seconds: int) -> Optional[str]:
```

关键代码：

```python
token = uuid.uuid4().hex
locked = redis_client.set(lock_key, token, nx=True, ex=ttl_seconds)
```

这里的关键点有三个：

1. `nx=True`
   - 表示只有 Key 不存在时才能设置成功
2. `ex=ttl_seconds`
   - 给锁加过期时间，防止死锁
3. `token`
   - 锁值不是简单写 `1`，而是随机字符串，用于校验锁归属

### 为什么锁值必须是 token

因为释放锁时必须确认：

- 这把锁是不是当前请求加的

否则会出现这种危险情况：

1. 请求 A 拿到锁
2. 锁过期
3. 请求 B 拿到新锁
4. 请求 A 执行释放锁
5. A 把 B 的锁误删了

所以需要用 token 做“锁归属校验”。

### 7.4 第二步：安全释放锁

函数：

```python
def release_lock(lock_key: str, token: str) -> None:
```

关键代码：

```python
redis_client.eval(REMOVE_LOCK_SCRIPT, 1, lock_key, token)
```

Lua 脚本逻辑：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 为什么必须用 Lua

因为“先 get，再 delete”不是原子操作。

在高并发下，中间可能被别的请求插入，导致误删别人的锁。Lua 脚本在 Redis 侧是原子执行的，能避免这个竞态问题。

### 7.5 第三步：拿到锁后做 double check

主流程代码：

```python
if lock_token:
    try:
        cache_status, cached_product = read_product_cache(cache_key)
        if cache_status == "HIT":
            return success(data=cached_product, message="从缓存获取成功")
```

### 为什么拿到锁后还要再查一次缓存

因为在当前请求抢锁的过程中，可能已经有其他请求先一步把缓存填好了。

如果不做 double check，就会出现：

- 明明缓存已经恢复了
- 当前请求还要多打一趟数据库

所以这一步是为了避免无意义的重复回源。

### 7.6 第四步：只有一个请求负责重建缓存

关键代码：

```python
product = query_product_from_db(product_id, db)
product_dict = serialize_product(product)
cache_product_detail(product_id, product_dict)
```

也就是说：

- 抢到锁的请求去查数据库
- 把查询结果写回 Redis
- 后面的请求就可以直接命中缓存

### 7.7 第五步：没抢到锁的请求等待缓存回填

函数：

```python
def wait_for_cache_rebuild(cache_key: str) -> Tuple[str, Optional[dict]]:
```

逻辑是：

1. 按 `product_cache_lock_wait_ms` 计算总等待时间
2. 每隔 `product_cache_lock_retry_ms` 轮询一次 Redis
3. 一旦发现缓存已经被填好，就直接返回

关键代码：

```python
deadline = time.time() + settings.product_cache_lock_wait_ms / 1000
retry_interval = max(settings.product_cache_lock_retry_ms, 1) / 1000

while time.time() < deadline:
    time.sleep(retry_interval)
    cache_status, cache_data = read_product_cache(cache_key)
    if cache_status != "MISS":
        return cache_status, cache_data
```

### 为什么这里要“等”，而不是所有请求直接查库

因为缓存击穿最怕的是：

- 一个热点 Key 过期
- 大量请求瞬间同时回源

如果后续请求愿意短暂等待，通常只需要第一个请求查库一次，后面的请求就能吃到缓存恢复后的结果。

这就是“单飞”或者“互斥重建”的核心思想。

### 7.8 第六步：等待超时后的兜底

主流程代码：

```python
print("等待缓存回填超时，执行数据库兜底查询")
product = query_product_from_db(product_id, db)
```

### 为什么还要保留兜底查库

因为分布式系统里不能假设：

- 拿到锁的那个请求一定成功

它可能发生：

- 进程崩溃
- 网络问题
- Redis 写失败
- 查询超时

如果没有兜底分支，后续请求可能会一直等，用户体验会非常差。

所以这里的设计目标是：

- 平时尽量只让一个请求回源
- 异常时允许少量兜底回源，保证接口最终可用

### 7.9 对应知识点总结

- Cache Breakdown / Cache Stampede
- Redis Distributed Lock
- Double Check
- SingleFlight 思想
- 可用性优先的降级设计

### 7.10 当前边界与后续优化

- 当前方案能显著削峰，但不能绝对保证“数据库永远只查一次”
- 在极端情况下，等待超时的兜底请求仍可能少量并发回源
- 如果后面流量更大，可以继续叠加：
  - 逻辑过期
  - 异步刷新
  - 本地单飞
  - 请求合并器

## 8. 缓存雪崩

### 8.1 什么是缓存雪崩

缓存雪崩指的是：

- 大量缓存 Key 在同一时间集中失效
- 请求在短时间内同时回源
- 数据库被瞬时冲垮

它和击穿的区别是：

- 击穿一般是一个热点 Key 出问题
- 雪崩一般是一批 Key 同时出问题

### 8.2 本项目的处理方案

本项目对雪崩的处理方案是：

- 正常缓存使用“基础 TTL + 随机抖动”
- 空值缓存也使用“基础 TTL + 随机抖动”

这样就能把缓存失效时间打散。

### 8.3 随机 TTL 的实现

函数：

```python
def ttl_with_jitter(base_seconds: int, jitter_seconds: int) -> int:
    return base_seconds + random.randint(0, max(jitter_seconds, 0))
```

### 为什么随机 TTL 能缓解雪崩

假设所有缓存都固定 3600 秒：

- 如果它们在同一段时间写入
- 那么 1 小时后也会在同一段时间一起过期

这会导致：

- 一批请求同时穿过缓存
- 一起去访问数据库

而加入随机抖动后：

- 有的 3605 秒过期
- 有的 3720 秒过期
- 有的 4180 秒过期

这样缓存失效时间就被打散了。

### 8.4 正常缓存中如何使用

函数：

```python
def cache_product_detail(product_id: str, product_dict: dict) -> None:
```

关键代码：

```python
redis_client.set(
    product_cache_key(product_id),
    json.dumps(product_dict),
    ex=ttl_with_jitter(
        settings.product_cache_ttl_seconds,
        settings.product_cache_ttl_jitter_seconds,
    ),
)
```

### 8.5 空值缓存中如何使用

函数：

```python
def cache_empty_product(product_id: str) -> None:
```

关键代码：

```python
redis_client.set(
    product_cache_key(product_id),
    PRODUCT_EMPTY_PLACEHOLDER,
    ex=ttl_with_jitter(
        settings.product_null_cache_ttl_seconds,
        settings.product_null_cache_ttl_jitter_seconds,
    ),
)
```

### 为什么空值缓存也要做随机 TTL

因为在恶意扫描或大量非法请求场景下：

- 不存在商品对应的空值 Key 也可能很多
- 如果这些空值缓存也同一时间过期
- 它们同样会再次集中回源数据库

所以空值缓存也不能忽视雪崩问题。

### 8.6 对应知识点总结

- Cache Avalanche
- TTL Jitter
- 批量失效分散化

### 8.7 当前边界与后续优化

随机 TTL 只能缓解雪崩，不能彻底消除所有批量失效问题。

如果后面场景更复杂，还可以进一步加：

- 多级缓存
- 缓存预热
- 逻辑过期
- 后台异步刷新
- 限流与熔断

## 9. 补充：为什么创建商品时要顺手预热缓存

在 `create_product` 中，当前项目做了下面这两步：

```python
add_product_id_to_index(new_product.id)
cache_product_detail(new_product.id, product_dict)
```

这样做的目的有两个：

1. 新商品一创建出来，就立刻进入商品 ID 索引
2. 新商品详情被提前写入缓存，减少第一次读取时的冷启动成本

这说明缓存治理不只是“读的时候怎么防御”，也包括“写的时候怎么配合维护辅助结构”。

---

## 10. 补充：为什么 Redis 异常时没有直接让接口失败

在多个函数里都能看到类似逻辑：

```python
except RedisError as exc:
    print(...)
    return "MISS", None
```

或者：

```python
except RedisError as exc:
    print(...)
    return True
```

这体现的是一种明确的工程取舍：

- 缓存层异常时，优先保证接口可用
- 尽量退化到数据库查询
- 不让 Redis 故障直接升级为用户可见的业务失败

这种策略的优点是：

- 服务可用性更高

代价是：

- 高并发时数据库压力会明显增加

所以这是一种“可用性优先”的设计，而不是“绝对性能最优”的设计。

## 11. 当前实现的不足与 TODO

虽然当前三类问题都已经做了基础治理，但还存在一些现实边界：

### 11.1 商品更新 / 删除场景还没补齐

当前商品服务主要实现了：

- 创建商品
- 查询商品

如果后续加入：

- 修改商品
- 下架商品
- 删除商品

那么必须同步处理：

- 商品详情缓存更新或失效
- 商品 ID 索引同步删除或修正
- 空值缓存误判清理

否则缓存治理本身会变成数据一致性风险来源。

### 11.2 商品 ID 索引当前是 Redis Set

当前实现的优点：

- 直观
- 好验证
- 零误判

但后面如果商品数量特别大，可以考虑：

- Bloom Filter
- 定时全量校准
- 消息队列异步同步索引

### 11.3 击穿治理仍然是“削峰”，不是“绝对只查一次”

当前已经尽量控制回源并发，但在极端情况下：

- 锁失败
- 等待超时
- Redis 抖动

仍可能出现少量兜底查库。

这不是 bug，而是当前工程策略中的有意权衡。

---

## 12. 最后总结

本项目处理这三个缓存问题的方式可以概括为：

### 缓存穿透

- 先查 Redis 商品 ID 索引
- 不存在则直接拦截
- 对不存在商品写入空值缓存

### 缓存击穿

- 缓存失效时先抢 Redis 分布式锁
- 只让一个请求回源查库并重建缓存
- 其他请求短暂等待缓存回填
- 等待失败时再兜底查库

### 缓存雪崩

- 正常缓存使用随机 TTL
- 空值缓存也使用随机 TTL
- 把失效时间打散，避免同一时刻集中回源

> 本项目在商品详情接口中，将缓存穿透、缓存击穿、缓存雪崩三类问题统一纳入一条读链路治理。  
> 穿透通过“存在性索引 + 空值缓存”解决，击穿通过“分布式锁 + 等待回填 + 兜底查库”解决，雪崩通过“随机 TTL”缓解。

这就是当前实现的核心设计。
