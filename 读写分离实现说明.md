# MySQL 读写分离实现说明

## 1. 目标

本次改造的目标不是把整个项目一次性改造成复杂的数据库中间件架构，而是实现一个课程作业里“可运行、可解释、可演示”的读写分离版本。

当前只对 `product_service` 做读写分离，原因如下：

- 商品服务最典型地包含“高频读、低频写”的特征
- 商品列表和商品详情非常适合作为读请求示例
- 创建商品接口天然是写请求，边界清晰
- 订单和库存场景对一致性要求更高，第一阶段不适合直接全部切到从库

最终效果是：

- `POST /api/products` 写主库
- `GET /api/products` 读从库
- `GET /api/products/{product_id}` 回源查库时读从库

---

## 2. 整体架构

这次读写分离涉及三层：

1. 基础设施层：MySQL 主从复制
2. 应用配置层：读库连接串 / 写库连接串
3. 业务代码层：根据请求类型选择不同 Session

逻辑关系如下：

```text
create_product
  -> write_db
  -> mysql-master

get_products
  -> read_db
  -> mysql-slave

get_product
  -> 先查 Redis
  -> 缓存未命中时 read_db
  -> mysql-slave
```

---

## 3. Docker Compose 层如何实现

本次新增了 3 个服务：

- `mysql-master`
- `mysql-slave`
- `mysql-replica-init`

对应文件：

- `docker-compose.yml`
- `mysql_readwrite/master/conf.d/master.cnf`
- `mysql_readwrite/slave/conf.d/slave.cnf`
- `mysql_readwrite/scripts/init-replica.sh`

### 3.1 主库 `mysql-master`

主库职责：

- 接收所有写请求
- 打开 binlog
- 作为从库复制的数据源

关键配置：

```ini
server-id = 101
log_bin = mysql-bin
binlog_format = ROW
gtid_mode = ON
enforce_gtid_consistency = ON
```

为什么这样配：

- `server-id`
  - 主从实例必须有不同身份编号
- `log_bin`
  - 没有 binlog，从库就没有可复制的变更流
- `ROW`
  - 行级复制更稳定，更适合业务表复制
- `GTID`
  - 比“文件名 + 偏移量”更容易维护，也更适合作业演示

### 3.2 从库 `mysql-slave`

从库职责：

- 承接读请求
- 通过复制追平主库
- 对应用保持只读

关键配置：

```ini
server-id = 102
relay_log = relay-bin
read_only = ON
super_read_only = ON
gtid_mode = ON
enforce_gtid_consistency = ON
```

为什么这样配：

- `read_only` / `super_read_only`
  - 防止应用误写从库
- `relay_log`
  - 暂存主库复制过来的事件
- `GTID`
  - 与主库保持同一套复制协议

### 3.3 复制初始化器 `mysql-replica-init`

这个容器只负责一件事：

- 在主库和从库都启动后，执行主从复制配置

对应脚本：

- `mysql_readwrite/scripts/init-replica.sh`

主要步骤：

1. 等待主库可连接
2. 等待从库可连接
3. 在主库创建复制用户 `repl`
4. 在从库执行 `CHANGE REPLICATION SOURCE TO`
5. 启动 `START REPLICA`
6. 打印 `SHOW REPLICA STATUS`

为什么要单独做成一个初始化容器：

- 复制关系不是镜像静态属性，而是“两个已启动实例之间”的动态编排
- 单独抽出来后，职责更清晰
- 日志里能直观看到复制是否建立成功

## 4. 配置层如何实现

对应文件：

- `product_service/config.py`

新增配置：

```python
write_database_url: Optional[str] = None
read_database_url: Optional[str] = None
```

保留兼容配置：

```python
database_url: str = ""
```

并新增两个 property：

```python
effective_write_database_url
effective_read_database_url
```

为什么这么做：

- 历史代码和测试脚本还在用 `DATABASE_URL`
- 新环境需要显式区分主库和从库
- 把兼容逻辑收敛到配置层，比散落在业务代码里更稳

读库的回退规则是：

- 优先 `READ_DATABASE_URL`
- 否则回退到 `WRITE_DATABASE_URL`
- 再不行回退到 `DATABASE_URL`

这样做的意义是：

- 本地单库调试仍可运行
- SQLite 测试仍可运行
- 主从环境也能正常启用读写分离

## 5. 数据库层如何实现

对应文件：

- `product_service/database.py`

### 5.1 双引擎

现在数据库层显式创建两个 engine：

```python
write_engine = create_engine(settings.effective_write_database_url)
read_engine = create_engine(settings.effective_read_database_url)
```

以及两个 Session：

```python
WriteSessionLocal = sessionmaker(...)
ReadSessionLocal = sessionmaker(...)
```

### 5.2 双依赖

新增：

```python
def get_write_db():
def get_read_db():
```

这一步很关键，因为读写分离真正落地的标志，不是“配置里有两个库”，而是“路由代码明确声明当前请求应该依赖哪台库”。

### 5.3 为什么只在写库上建库和建表

代码里现在只对写库执行：

```python
if not database_exists(write_engine.url):
    create_database(write_engine.url)
```

以及在 `main.py` 中：

```python
models.Base.metadata.create_all(bind=write_engine)
```

原因是：

- 主库必须是唯一事实源
- 从库应该只通过复制追平主库
- 如果应用自己在从库直接建表或建库，会破坏复制拓扑的前提

### 5.4 启动时等待从库表结构同步

新增函数：

```python
wait_for_read_replica_tables_ready(...)
```

为什么要做这一步：

- 商品服务启动时会在主库建表
- 但从库同步这些 DDL 不是绝对瞬时完成
- 如果应用启动完成后立刻接收读请求，而从库表结构还没追平，第一次读就可能失败

这一步是在启动期主动等待从库追平，把问题尽量留在系统内部，而不是暴露给第一个用户请求。

## 6. 路由层如何实现

对应文件：

- `product_service/api/routes.py`

### 6.1 写接口走主库

`create_product` 现在使用：

```python
db: Session = Depends(database.get_write_db)
```

这表示：

- 创建商品只允许走主库
- 不会误把写请求发到从库

### 6.2 读接口走从库

`get_products` 使用：

```python
db: Session = Depends(database.get_read_db)
```

`get_product` 也使用：

```python
db: Session = Depends(database.get_read_db)
```

需要注意：

- `get_product` 先查 Redis
- 只有缓存未命中时才真正访问从库

所以如果你要演示“详情接口确实读从库”，最好先清 Redis，或者用未命中的商品场景。

### 6.3 为什么列表接口特别适合验证读写分离

因为它：

- 天然是纯读请求
- 当前没有 Redis 缓存短路
- 每次都会真正打数据库

所以在答辩演示时，`GET /api/products` 是最推荐的验证接口。

## 7. 如何在代码中验证读写分离效果

对应函数：

```python
def log_db_routing(db: Session, operation: str) -> None:
```

这个函数会执行：

```sql
SELECT @@hostname, @@server_id, @@read_only;
```

然后打印日志：

```text
[DB ROUTING] operation=create_product, role=WRITE_PRIMARY, host=..., server_id=..., read_only=0
[DB ROUTING] operation=get_products, role=READ_REPLICA, host=..., server_id=..., read_only=1
```

为什么这一层很重要：

- 读写分离如果只写在 docker-compose 里，老师看不到运行时证据
- 把命中的实例角色直接打印出来，才能证明代码真的走到了不同数据库

调用：


预期日志：

## 9. WARNING 与边界

### WARNING 1：从库存在复制延迟

这意味着：

- 刚刚写入主库的数据
- 在非常短的时间内
- 从库可能还没同步到

所以“刚写完马上读”在极端情况下可能读不到最新数据，这是读写分离的典型副作用，不是实现错误。

### WARNING 2：缓存会掩盖数据库分离效果

商品详情接口有 Redis 缓存。

如果缓存命中：

- 请求根本不会打数据库
- 你也就看不到主从区别

所以演示数据库读写分离时，优先用 `GET /api/products` 更直观。

### WARNING 3：当前只改造了商品服务

这是有意为之。

原因是：

- 商品服务适合做“读多写少”的演示
- 订单、库存更依赖强一致，不适合第一版直接切从库

这不是没做完，而是阶段性架构取舍。

## 11. 总结

本次读写分离实现的核心点可以概括为：

1. Docker Compose 中增加主库、从库和复制初始化器
2. 商品服务配置层支持读库 / 写库双连接
3. 数据库层显式拆分 `get_write_db` 和 `get_read_db`
4. 写接口走主库，读接口走从库
5. 日志打印数据库角色，作为运行时证据

> 本项目在商品服务中实现了 MySQL 主从复制和代码级读写路由，创建商品固定写主库，商品列表与详情回源固定读从库，并通过运行时日志打印当前命中的数据库角色，验证读写分离已经真正生效。
