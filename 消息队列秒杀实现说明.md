# 消息队列秒杀实现说明

## 1. 改造目标

这部分改造解决的是“秒杀流量不能直接同步打数据库”的问题。

原来的下单链路是：

1. 请求进入订单服务
2. 同步查询商品服务
3. 同步调用库存服务锁库存
4. 同步写订单库

这个方案在普通 CRUD 演示里足够直观，但在秒杀场景下会有两个明显问题：

1. 每个请求都直接打到数据库和下游服务，洪峰来时数据库最先扛不住。
2. 下单入口耗时长，请求线程被同步链路占住，系统吞吐会很快掉下来。

所以这次改造成了：

1. Redis 负责高并发库存闸门和用户幂等拦截
2. Kafka 负责异步削峰
3. 订单服务后台消费者负责真正创建订单
4. 库存服务负责数据库最终扣减与回补

这样做的核心不是“把技术名词堆上去”，而是把最脆弱的同步写链路拆开：

- Redis 擅长处理热点 Key 的高并发原子操作
- Kafka 擅长把瞬时洪峰摊平到后端消费者慢慢处理
- MySQL 只承担最终持久化，而不直接暴露在抢购洪峰入口

## 2. 当前实现的整体流程

### 2.1 下单入口

入口文件：

- `order_service/api/routes.py`
- `order_service/flash_sale.py`

用户调用 `POST /api/orders` 后，系统按下面顺序执行：

1. 订单服务先生成雪花订单 ID
2. 调用 Redis Lua 脚本，原子完成三件事：
   - 判断库存是否足够
   - 判断当前用户是否已经抢过同一商品
   - 库存减 1，并把“用户已抢购”标记写入 Redis
3. 如果 Redis 中没有该商品库存 Key，订单服务会调用库存服务预热缓存
4. Redis 预扣成功后，把订单事件写入 Kafka
5. Kafka 确认接收成功后，立即返回“排队中”

这里最关键的设计点是：

- 返回给前端的成功，不代表订单已经写入 MySQL
- 它只代表“库存闸门已经通过，消息已经成功进入队列”

这样用户不会一直卡在同步请求里等数据库写完，而系统也不会在洪峰时把压力直接压到下游。

### 2.2 异步创建订单

消费端文件：

- `order_service/order_consumer.py`
- `order_service/main.py`

订单服务启动时会拉起 Kafka 消费线程。消费者拿到消息后按顺序执行：

1. 先检查这条订单是否已经创建成功
2. 再检查数据库里是否已经存在同一用户同一商品的订单
3. 调用商品服务查询秒杀价，拿到订单金额快照
4. 在订单库中创建 `PROCESSING` 状态的订单
5. 调用库存服务的秒杀最终扣减接口
6. 库存扣减成功后，把订单状态更新成 `PENDING_PAYMENT`

这里故意把订单先写成 `PROCESSING`，而不是一上来就写成最终成功状态，原因是：

- 异步链路不是一个原子大事务
- 订单创建和库存最终扣减分属不同服务
- 中间任一步失败，都必须有一个可观测的中间态，方便补偿和排障

如果消费者在“订单已落库但库存确认还没完成”时崩溃，下一次重复消费时还能基于 `PROCESSING` 状态继续补全后续步骤。

### 2.3 库存最终一致性

库存侧文件：

- `inventory_service/api/routes.py`
- `inventory_service/models.py`

库存服务新增了一张 `flash_sale_inventory_records` 表，用来记录每个 `order_id` 的库存最终扣减结果。

它解决的是一个非常典型的问题：

- Kafka 是至少一次投递
- 至少一次投递就意味着同一条消息可能被消费两次
- 如果库存服务没有自己的幂等记录，就无法判断“这笔库存我到底扣过没有”

所以当前库存服务的秒杀接口策略是：

1. `commit` 时先查 `order_id`
2. 如果已经扣过，直接返回成功
3. 如果没扣过，才锁住库存行并真正减少数据库库存
4. `restore` 时也是同样逻辑，避免重复回补

这就是“幂等不是避免重复消息，而是重复消息来了结果仍然正确”的具体落地。

## 3. 关键问题是怎么解决的

### 3.1 削峰填谷

实现位置：

- `order_service/api/routes.py`
- `order_service/flash_sale.py`
- `order_service/order_consumer.py`

做法：

1. 抢购请求进入后不直接写数据库
2. 只做 Redis 原子预扣和 Kafka 投递
3. 数据库写操作放到消费者后台串行/批量承接

为什么这能削峰：

- 因为入口线程不再被数据库 IO 长时间占住
- 因为瞬时 1000 个请求不再等价于瞬时 1000 次数据库写
- 而是先变成 1000 条消息，再由消费者按处理能力平滑消费

### 3.2 幂等性

实现位置：

- `order_service/flash_sale.py`
- `order_service/models.py`
- `inventory_service/models.py`
- `inventory_service/api/routes.py`

当前用了三层兜底：

1. Redis 用户购买标记
   - Key 形式：`flashsale:user-order:{product_id}:{user_id}`
   - 在入口层挡住绝大多数重复下单

2. 订单表唯一约束
   - `UniqueConstraint("user_id", "product_id")`
   - 防止 Redis 丢数据、重启或并发异常时，重复订单落库

3. 库存扣减记录表
   - 以 `order_id` 为主键
   - 防止 Kafka 重复消费导致数据库库存被扣两次

这三层为什么都要有：

- 只靠 Redis，挡不住缓存丢失后的重复落库
- 只靠数据库唯一约束，入口高并发时会让数据库承受大量无意义竞争
- 只靠订单幂等，挡不住库存侧的重复最终扣减

秒杀系统里“幂等”不是一个点，而是一条链路上多层一起做。

### 3.3 数据一致性

实现位置：

- `order_service/order_consumer.py`
- `inventory_service/api/routes.py`
- `order_service/api/routes.py`

当前一致性策略是“本地落库 + 失败补偿”，而不是追求跨服务强一致大事务。

处理方式：

1. Redis 先预扣库存，保证高并发入口不超卖
2. Kafka 异步消费创建订单
3. 库存服务最终扣减数据库库存
4. 如果中间任一步失败：
   - 订单状态标记为 `FAILED`
   - Redis 可售库存自动回补
   - Redis 用户购买标记清理，允许用户重新秒杀

为什么这里不直接做分布式事务：

- 课程项目里目标是展示“高并发下的可行治理方案”
- 秒杀链路比强一致更重要的是“先保证不超卖，再保证可恢复”
- 引入分布式事务会把作业复杂度和演示成本拉得很高，反而不利于解释

当前方案的重点是：

- 最终库存不会因为重复消费被多扣
- 订单失败时 Redis 库存会补回去
- 用户能通过订单状态看到异步处理结果

## 4. 雪花订单 ID 是怎么做的

实现位置：

- `order_service/flash_sale.py`

订单 ID 没有继续用数据库自增，也没有沿用原先的 UUID，而是换成了雪花算法。

原因有三个：

1. 秒杀请求一进入系统，就需要马上返回一个可查询的订单号
2. 如果订单号依赖数据库生成，就会把数据库强依赖重新拉回入口线程
3. 雪花 ID 近似有序，日志排查和后续按时间维度分析更方便

当前生成器包含：

- 时间戳
- worker_id
- 毫秒内序列号

同时加入了“时钟回拨时等待到安全时间点”的保护，避免生成重复 ID。

## 6. 如何启动和验证

### 6.1 启动

如果你之前跑过旧版本，建议先清理旧容器和旧数据卷：

```powershell
docker compose down -v
docker compose up -d --build
```

为什么建议这样做：

- 订单表和库存表都新增了字段/表
- 课程项目当前没有引入 Alembic 这类正式迁移工具
- 用干净环境验证最省时间，也最不容易被历史脏数据干扰

### 6.2 初始化库存

先通过库存服务给某个商品加库存：

```http
POST /api/inventory
{
  "product_id": "商品ID",
  "total_stock": 20
}
```

这一步除了写数据库，还会同步把 Redis 秒杀库存加上去。

### 6.3 发起秒杀

调用：

```http
POST /api/orders
Authorization: Bearer <token>
{
  "product_id": "商品ID"
}
```

你会先拿到：

```json
{
  "code": 200,
  "message": "秒杀请求已成功进入队列",
  "data": {
    "order_id": "xxx",
    "status": "QUEUED",
    "message": "秒杀请求已入队，系统正在异步创建订单，请稍后查询订单状态"
  }
}
```

### 6.4 查询订单状态

然后马上查询：

```http
GET /api/orders/{order_id}
```

你会看到订单状态从下面几个状态中流转：

- `QUEUED`
- `PROCESSING`
- `PENDING_PAYMENT`
- `FAILED`
- `PAID`
- `CANCELLED`

### 6.5 看日志验证

建议同时观察：

```powershell
docker compose logs -f order-service
docker compose logs -f inventory-service
docker compose logs -f kafka
```

重点看：

1. 订单服务是否打印 Kafka 消费日志
2. 库存服务是否打印秒杀库存扣减/回补日志
3. 用户重复下单时是否被直接拦截
4. 库存扣完后是否返回“库存不足”

## 7. 改造总结

1. 普通同步下单为什么不适合秒杀
2. Redis 先挡住热点库存和重复下单
3. Kafka 把洪峰流量削平
4. 消费者异步创建订单
5. 库存服务用幂等记录表保证最终扣减只发生一次
6. 失败时回补 Redis 库存，保证不会“扣了库存却没单”
7. 订单号提前生成，所以用户能立即拿到查询凭证

## 8. WARNING 与 TODO

### WARNING

- 当前 Kafka 消费者是嵌在 `order_service` Web 进程里的，适合课程项目演示，但不适合生产环境大规模水平扩容。
- 当前 Redis 库存预热在“干净启动、无积压消息”的课堂场景里是安全的；如果未来存在 Redis 丢数据但 Kafka 仍有大量未消费消息，不能简单用数据库库存直接覆盖 Redis。
- 当前项目没有引入正式数据库迁移工具；如果你切换过旧版本，验证前最好 `docker compose down -v`。
